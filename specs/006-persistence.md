# 006 — 任务状态持久化与一致性

- **状态**: proposed
- **优先级**: 中
- **创建**: 2026-06-10
- **依赖**: 无

## 背景 / 动机

任务列表来自对 `~/.claude/projects` 的实时扫描，而 done/skip/defer/dismiss 等人工操作是另一份状态（`data/tasks.json` / `history.json`）。需要确认：用户操作是否可靠落盘、重启后是否一致、扫描结果与人工状态如何合并去重。

## 目标

明确并保证任务状态的持久化与扫描结果的一致合并，重启不丢人工操作。

### 非目标

- 不引入数据库（继续用 JSON 文件 + 原子写）。

## 需求

- 作为用户，我标记「完成/跳过/延后」后重启 Commander，状态仍在。
- 同一会话不因重新扫描而「复活」已处理的任务。

## 验收标准

- [ ] done/skip/defer/dismiss 操作原子落盘（tmp+rename）
- [ ] 重启后人工状态恢复，已处理任务不复现
- [ ] 扫描结果与人工状态按稳定 key（会话 id？工作目录？）合并去重
- [ ] `data/` 已在 .gitignore（含本机路径，不入库）—— 已确认

## 技术方案

- 审计 `src/server/store.js` 的写入路径与 `scanner.js` 的合并逻辑，明确 key 与冲突规则。
- 先写一份「当前实际行为」的核对，再决定要不要改。

## 风险 / 待定

- 合并 key 的选择影响去重正确性（会话 id 会变？工作目录会重名？）需核实。
