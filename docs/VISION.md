# Commander Vision — 统一工作入口

## 洞察

AI 时代的工作方式正在发生根本变化：

1. **工作碎片化**：一个人同时在 5+ 个 AI 会话、GitLab、IM、文档、部署面板之间切换
2. **上下文切换是最大的效率杀手**：不是 AI 不够快，是人在工具之间切换太多
3. **每个人都需要一个"作战指挥台"**：看到所有工作的全局状态，在一个地方处理大部分事情

## 核心理念

**大部分工作直接在 Dashboard 处理，少数需要深入的才跳转到专用工具。**

不是又一个"聚合器"或"通知中心"。是一个**能直接行动的工作面板**：
- 看到 AI 完成了 → 直接看结果、给反馈，不用切窗口
- 看到 MR 需要 review → 直接看 diff、approve，不用打开 GitLab
- 看到消息需要回复 → 直接回复，不用切到 IM
- 只有需要深度处理的，才通过链接跳转到专用工作区

## 演进路径

```
Phase 1 (MVP)     → AI 多实例指挥台
                     解决自己的痛点：Claude Code 会话管理

Phase 2            → + 工具集成层
                     GitLab MR/Pipeline、IM 消息、部署状态
                     通过 MCP/插件接入，不硬编码

Phase 3            → + 内联操作
                     不只看状态，能直接在面板里处理
                     approve MR、回复消息、触发部署

Phase 4            → + 团队协作
                     团队共享面板，看到整个团队的工作状态
                     "小红书项目：小明在写前端，AI 在跑测试，等你 review"
```

## 关键设计原则

### 1. 插件化集成，不硬编码

每种工具通过插件接入，统一的事件格式：

```
┌──────────────────────────────────────┐
│          Commander Dashboard          │
├──────────────────────────────────────┤
│            统一事件总线               │
├────┬────┬────┬────┬────┬────────────┤
│ CC │ GL │ IM │ CI │ Doc│  ...       │
│    │    │    │    │    │            │
│ Claude GitLab Lark Deploy Notion   │
│ Code                               │
└────┴────┴────┴────┴────┴────────────┘
```

### 2. 渐进式深入

```
Dashboard 面板（全局视图）
  ↓ 需要更多细节
内联展开（在面板内看详情）
  ↓ 需要专用工具
一键跳转（打开 GitLab/IDE/IM）
```

### 3. 面向"事件流"而不是"工具"

传统方式：按工具组织（打开 GitLab → 看 MR，打开 IM → 看消息）
Commander：按事件流组织（"有什么需要我处理的？"→ 逐条处理，不管来源）

## 差异化

市面上的类似工具：
- **Linear/Jira**：项目管理，但不管 AI 实例和实时工具状态
- **Raycast**：快速启动器，但不是持续监控面板
- **tmux/screen**：终端多路复用，但没有语义化状态
- **各种 Dashboard**：展示型为主，不能直接操作

Commander 的独特价值：
- **AI-native**：把 AI 实例当作一等公民，不是附加功能
- **可操作**：不只看状态，能直接处理
- **个人优先**：先解决个人效率，再扩展到团队

## 潜在商业模式

- 核心开源（CLI + 基础插件）
- Pro 版：团队面板、高级集成（企业 GitLab/Jira/Slack）、历史分析
- 企业版：私有部署、SSO、审计日志

## 目标用户

Phase 1：像我一样同时跑多个 Claude Code 的重度 AI 用户
Phase 2：用 AI 工具 + 传统工具混合工作的工程师
Phase 3：整个工程团队
