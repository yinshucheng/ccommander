# Commander 团队版需求 — AI-native 工程效率平台

> 面向团队沟通的版本，用于和同事/老板讨论可行性

---

## 一句话

**工程师的统一作战面板**：把散落在 Claude Code、GitLab、大象、CI/CD 里的工作状态汇聚到一个界面，大部分直接处理，少数跳转深入。

## 我们观察到的问题

### 工程师的一天（现状）

```
09:30  打开大象，20条消息要回复
09:45  打开 GitLab，3个 MR 等 review
10:00  启动 Claude Code 写代码
10:20  切到大象回消息，打断了
10:30  回来发现 Claude Code 5分钟前就完成了，一直在等我
10:35  另一个 Claude Code 实例卡了15分钟，我不知道
10:40  GitLab pipeline 失败了，没注意到
11:00  老板问"那个功能做完了吗"，我在5个标签页里翻找进度
```

**核心问题：不是工具不够多，是人在工具之间切换太多。**

每次切换 = 丢失上下文 + 重新加载 + 思维中断。一天切换几十次，真正深度工作的时间可能不到 3 小时。

### AI 让问题加剧了

AI 工具的使用方式是"发出去等结果"，天然产生并发：
- 同时开 3-5 个 Claude Code 实例
- 每个都在不同项目/任务上
- 完成时间不确定，但都需要人审阅
- **人成了瓶颈，不是 AI**

### 三级深入：面板 → 内联 → 跳转

| 层级 | 操作 | 场景 |
|------|------|------|
| **面板** | 一眼扫全局 | "有什么需要我处理的？" |
| **内联展开** | 在面板内看详情、直接操作 | 看 MR diff 摘要、回复消息、查看 AI 输出 |
| **跳转** | 打开专用工具 | 复杂 code review、调试、长对话 |

**80% 的工作在面板和内联层处理完，只有 20% 需要跳转。**

## 技术架构

### 插件化设计

不硬编码任何工具，通过统一接口接入：

```
Commander Core（事件总线 + UI）
    ↑ 统一事件格式
    │
    ├── claude-code-plugin     ← Hook 采集，已有原型
    ├── gitlab-plugin          ← MR/Pipeline/Issue 状态
    ├── lark-plugin            ← 大象消息
    ├── ci-plugin              ← 部署/构建状态
    └── custom-plugin          ← 团队自定义
```

每个插件只需实现：
1. **pull**：定期拉取状态
2. **push**：实时推送事件（通过 webhook/hook）
3. **action**：定义可执行的操作（approve MR、回复消息等）

### 美团内部集成点

| 系统 | 接入方式 | 提供的事件 | 可执行操作 |
|------|---------|-----------|-----------|
| Claude Code / OpenClaw | Hook 脚本 | 会话状态变更 | 跳转/恢复会话 |
| GitLab | API + Webhook | MR 状态、Pipeline、review 请求 | Approve、Merge、评论 |
| 大象 IM | Bot API | 消息、@提醒 | 回复消息 |
| CI/CD | API + Webhook | 构建/部署状态 | 触发重试、查看日志 |
| Ones/Jira | API | 任务状态变更 | 更新状态 |

## 价值量化

### 对个人

| 指标 | 现状 | 目标 |
|------|------|------|
| 日均上下文切换次数 | ~50 次 | <15 次 |
| AI 等待响应平均时长 | 未知（经常 5min+） | <1min |
| "翻找某个会话/MR"的时间 | 每次 1-3 min | 即时 |

### 对团队

| 指标 | 价值 |
|------|------|
| AI 工具利用率可见 | 老板能看到团队实际在用 AI 做什么，推进 AI-native 有数据支撑 |
| 协作透明度 | "谁在做什么、什么状态" 不用问，看面板就知道 |
| 新人上手 | 新人一看面板就知道团队在做什么，不用到处问 |

## FAQ

**Q：和现有的 XXX 工具有什么区别？**
A：现有工具是"按工具组织"（打开 GitLab 看 MR，打开大象看消息），Commander 是"按事件流组织"（有什么需要我处理的？不管来源）。而且 AI 实例管理是全新的，没有现成工具做这个。

**Q：为什么不做成浏览器插件 / Chrome Extension？**
A：可以做，但终端 TUI 对工程师更友好，启动快、无干扰。后续可以同时提供 Web 版和 TUI 版。

**Q：会不会又多了一个要切换的工具？**
A：Commander 的目标恰恰是减少切换。它不是第 N+1 个工具，而是把 N 个工具的信息收拢到 1 个入口。如果每天在 Commander 里处理 80% 的事情，就只需要"Commander + 偶尔跳转"两个动作。

**Q：数据安全？**
A：Commander 跑在本地，不上传任何数据。团队版可以走内网部署。插件读取的是已有系统的数据，不创建新的数据通道。
